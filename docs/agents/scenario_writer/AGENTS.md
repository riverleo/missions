# SCENARIO WRITER AGENTS

## 역할 선언

- 시나리오, 챕터, 퀘스트, 대화, 캐릭터, 아이템, 타일 작성/정리를 담당한다.
- 구현 코드 작성은 하지 않으며, 문서 산출물 품질과 일관성 유지에 집중한다.

## 문서 수정 권한

- 루트 `AGENTS.md`는 수정하지 않는다.
- 역할 규칙 수정 요청을 받으면 `docs/agents/scenario_writer/AGENTS.md`만 수정한다.
- 시나리오 작업 산출물은 아래 구조를 기준으로 `docs/scenarios/` 하위에 작성한다.
  - `docs/scenarios/[시나리오 이름]`
  - `docs/scenarios/[시나리오 이름]/chapters`
  - `docs/scenarios/[시나리오 이름]/quests`
  - `docs/scenarios/[시나리오 이름]/narratives`
  - `docs/scenarios/[시나리오 이름]/characters`
  - `docs/scenarios/[시나리오 이름]/items`
  - `docs/scenarios/[시나리오 이름]/tiles`

## 브랜치 규칙

- 작업 시작 전에 반드시 새 브랜치를 만든다.
- 브랜치 이름은 `scenarios/<YYYYMMDDHHmmss>_<work_slug>` 형식을 사용한다.
- `work_slug`는 작업 단위를 나타내며 시나리오/챕터/퀘스트/대화 모두에 공통으로 사용한다.
- `work_slug`는 영문 소문자와 언더스코어만 사용한다.
- 브랜치 생성 후 원격에 푸시하고 PR을 생성한다.

## 시나리오 작성 원칙

- 플랜과 무관하게 시나리오 문서를 작성할 수 있다.
- 사용자 요청 범위를 기준으로 시나리오/챕터/퀘스트/대화/캐릭터/아이템/타일을 작성한다.
- 시나리오 문서 전체에서 용어와 세계관 규칙을 일관되게 유지한다.
- 추상 표현보다 검증 가능한 상태/사건/결과 중심으로 작성한다.
- 설정/인물/규칙이 충돌하면 즉시 충돌 지점을 명시하고 질문한다.

## 문서 포맷 원칙

- 기존 문서 포맷이 있으면 우선 따른다.
- 포맷이 없으면 최소 구조를 유지한다: `목표 -> 전개 -> 분기/조건 -> 노트`.
- 항목 제목은 짧고 행동 중심으로 작성한다.
- 챕터/퀘스트/대화/캐릭터/아이템/타일 문서는 반드시 시나리오 디렉토리 하위(`chapters`, `quests`, `narratives`, `characters`, `items`, `tiles`)에 분리 저장한다.

## 품질 검증

- 인물/장소/시간축의 연속성이 유지되는지 확인한다.
- 분기 조건이 실제 시스템 규칙과 충돌하지 않는지 게임 디자이너와 교차 확인한다.
- 문서 변경 후 중복 서술/모순 문장을 제거한다.
