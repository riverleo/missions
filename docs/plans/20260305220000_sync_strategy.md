# 데이터 싱크 전략 수립

## 목표

게임의 세 가지 저장소(컨텍스트/로컬 스토리지/데이터베이스) 간 싱크 전략을 명확히 정의하고 구현한다.

1. **컨텍스트 → 로컬 스토리지**: 매 틱 완료 시 자동 저장하여 새로고침 시 데이터 유실을 방지한다.
2. **로컬 스토리지 → 데이터베이스**: 월드 상태를 JSON 스냅샷으로 DB에 저장한다.

### 현재 문제

- 스토어 업데이트는 실시간이지만 localStorage 저장은 수동(`⌘⇧S`)이라 새로고침 시 데이터가 유실된다.
- 게임 액션(아이템 줍기/사용, 캐릭터 이동, 욕구 변화)이 런타임 상태만 변경하고 영속 데이터에는 반영되지 않는 구간이 존재한다.
- `world_character_id` 버그처럼 런타임 상태와 영속 데이터의 불일치로 인한 버그가 반복 발생한다.

### 싱크 방향

**컨텍스트 → 로컬 스토리지 (결정됨)**

매 틱 완료 시점에 `saveAllEntities()` + `save()`를 호출한다. 현재 `TICK_INTERVAL`은 100ms이므로 초당 10회 저장이 발생한다.

**로컬 스토리지 → 데이터베이스 (결정됨: JSON 스냅샷)**

기존 `worlds` 테이블에 `snapshot jsonb` 컬럼을 추가하여 월드 전체 상태를 저장한다. 게임은 미션/할일 관리의 인터랙티브 표현 레이어이므로 어뷰징 차단보다 단순성을 우선한다. 다른 사람 월드 방문은 `worlds` 테이블의 `select` 정책이 이미 public이므로 스냅샷을 읽기 전용으로 로드하여 바로 구현 가능하다.
