# 데이터 싱크 전략 수립

## 목표

게임의 세 가지 저장소(컨텍스트/로컬 스토리지/데이터베이스) 간 싱크 전략을 명확히 정의하고 구현한다.

1. **컨텍스트 → 로컬 스토리지**: 매 틱 완료 시 자동 저장하여 새로고침 시 데이터 유실을 방지한다.
2. **로컬 스토리지 → 데이터베이스**: 월드 상태를 JSON 스냅샷으로 DB에 저장한다.

### 현재 문제

- 스토어 업데이트는 실시간이지만 localStorage 저장은 수동(`⌘⇧S`)이라 새로고침 시 데이터가 유실된다.
- 게임 액션(아이템 줍기/사용, 캐릭터 이동, 욕구 변화)이 런타임 상태만 변경하고 영속 데이터에는 반영되지 않는 구간이 존재한다.
- `world_character_id` 버그처럼 런타임 상태와 영속 데이터의 불일치로 인한 버그가 반복 발생한다.

### 싱크 방향

**컨텍스트 → 로컬 스토리지 (결정됨)**

매 틱 완료 시점에 `saveAllEntities()` + `save()`를 호출한다. 현재 `TICK_INTERVAL`은 100ms이므로 초당 10회 저장이 발생한다.

**로컬 스토리지 → 데이터베이스 (결정됨: JSON 스냅샷)**

기존 `worlds` 테이블에 `snapshot jsonb` 컬럼을 추가하여 월드 전체 상태를 저장한다. 게임은 미션/할일 관리의 인터랙티브 표현 레이어이므로 어뷰징 차단보다 단순성을 우선한다. 다른 사람 월드 방문은 `worlds` 테이블의 `select` 정책이 이미 public이므로 스냅샷을 읽기 전용으로 로드하여 바로 구현 가능하다.

#### 리팩토링 — 단계 4: WorldSnapshot 재설계 및 WorldData 제거

> 단계 3 완료 후 수행. WorldSnapshot을 엔티티 데이터 전용으로 재설계하고, WorldData를 제거한다.

**현재 문제:**

1. `WorldSnapshot`에 `player`와 `playerScenario`가 포함되어 있다. 이 데이터는 이미 `usePlayer()` 스토어와 DB `players`/`player_scenarios` 테이블에 존재하므로 **순수 중복**이다.
2. `WorldSnapshot.worlds: Record<WorldId, WorldData>` 구조는 불필요한 중첩이다.
3. `WorldData`는 `Omit<World, 'snapshot'> & 엔티티 레코드`인데, World 메타는 DB `World` 타입이 이미 담당한다. `WorldData`가 World 메타를 중복 포함할 이유가 없다.
4. `buildSnapshot()`이 `useCurrent()`에 의존하여 player/playerScenario를 읽는데, 테스트 월드에서는 derived store가 비어있어 에러가 발생한다.

**수정 방향:**

- `WorldData` 제거.
- `WorldSnapshot` = **엔티티 데이터만**. World 메타(id, name, terrain_id 등)는 DB `World` 타입이 담당하고, `World.snapshot`을 `WorldSnapshot`으로 캐스팅하여 엔티티에 접근한다.

  ```typescript
  // core.ts — 엔티티 데이터만. World 메타 포함하지 않음.
  interface WorldSnapshot {
      worldCharacters: Record<WorldCharacterId, WorldCharacter>;
      worldCharacterNeeds: Record<WorldCharacterNeedId, WorldCharacterNeed>;
      worldBuildings: Record<WorldBuildingId, WorldBuilding>;
      worldBuildingConditions: Record<WorldBuildingConditionId, WorldBuildingCondition>;
      worldItems: Record<WorldItemId, WorldItem>;
      worldTileMap?: WorldTileMap;
  }
  ```

- `TestWorldSnapshot extends WorldSnapshot`. 테스트 월드도 동일한 엔티티 스냅샷이며, 저장소가 localStorage일 뿐 같은 용도이다.

  ```typescript
  // hooks.ts
  interface TestWorldSnapshot extends WorldSnapshot {
      player: Player;
      playerScenario: PlayerScenario;
      open: boolean;
      openPanel: boolean;
      selectedTerrainId?: TerrainId;
      modalScreenVector: ScreenVector;
      debug: boolean;
  }
  ```

- 런타임 데이터 접근: `World` + `world.snapshot as WorldSnapshot`으로 조합. 별도 통합 타입 불필요.

- 리팩토링 요구사항 6 — **useWorld CRUD를 create/delete 패턴으로 변경**:
  - 단계 1에서 추가한 `set*`/`remove*` 함수는 완성된 객체를 그대로 받는 단순 setter이므로 부족하다.
  - `create*`(Pick<Entity, ...>)` 패턴으로 변경하여 UUID 생성, 기본값(timestamps, player_id, scenario_id 등)을 훅 내부에서 처리한다.
  - context 함수는 `useWorld().create*(...)`를 호출하여 데이터를 받고, 엔티티 라이프사이클(addToWorld/removeFromWorld)만 담당한다.
  - 변경 대상:
    - `setWorldCharacter` → `createWorldCharacter(pick)`: UUID, player_id, scenario_id, timestamps 내부 처리
    - `setWorldBuilding` → `createWorldBuilding(pick)`: 동일
    - `setWorldItem` → `createWorldItem(pick)`: 동일
    - `setWorldTileMap` → `createWorldTileMap(pick)`: 동일
    - `setWorldCharacterNeeds` → `createWorldCharacterNeeds(worldCharacterId, characterNeeds)`: need 목록 생성 내부 처리
    - `setWorldBuildingConditions` → `createWorldBuildingConditions(worldBuildingId, buildingConditions)`: condition 목록 생성 내부 처리
  - `remove*` / `delete*` 네이밍도 통일한다 (`deleteWorldCharacter`, `deleteWorldBuilding` 등).
- 리팩토링 요구사항 7 — **WorldSnapshot 데이터 중복 제거**:
  - 근본 문제: `WorldSnapshot`이 `player`, `playerScenario`, `worlds: Record<WorldId, WorldData>` 세 필드를 가지는데, player/playerScenario는 이미 각각 `players`/`player_scenarios` 테이블과 `usePlayer()` 스토어에 존재. `worlds` 래핑도 스냅샷은 항상 단일 월드 데이터이므로 무의미한 중첩 (DB row당 1개 world, 테스트 월드도 앱 내 1개).
  - 영향: `buildSnapshot()`이 player/playerScenario를 읽기 위해 `useCurrent()`에 의존 → 테스트 월드에서 derived store가 비어있어 `requireCurrentContext()` 에러 발생 → 이 에러의 근본 원인이 WorldSnapshot 설계 자체.
  - `fetch()`에서 `snapshot?.worlds?.[world.id]`로 이중 접근하는 것도 이 잘못된 구조 때문.
  - 해결: `WorldData` 제거. `WorldSnapshot` = 엔티티 데이터만 (World 메타 포함하지 않음). World 메타는 DB `World` 타입이 담당. `World.snapshot`을 `WorldSnapshot`으로 캐스팅. `TestWorldSnapshot extends WorldSnapshot`.
