# 데이터 싱크 전략 수립

## 목표

게임의 세 가지 저장소(컨텍스트/로컬 스토리지/데이터베이스) 간 싱크 전략을 명확히 정의하고 구현한다.

1. **컨텍스트 → 로컬 스토리지**: 매 틱 완료 시 자동 저장하여 새로고침 시 데이터 유실을 방지한다.
2. **로컬 스토리지 → 데이터베이스**: 저장 전략을 결정하고 구현한다. (방향 논의 필요)

### 현재 문제

- 스토어 업데이트는 실시간이지만 localStorage 저장은 수동(`⌘⇧S`)이라 새로고침 시 데이터가 유실된다.
- 게임 액션(아이템 줍기/사용, 캐릭터 이동, 욕구 변화)이 런타임 상태만 변경하고 영속 데이터에는 반영되지 않는 구간이 존재한다.
- `world_character_id` 버그처럼 런타임 상태와 영속 데이터의 불일치로 인한 버그가 반복 발생한다.

### 싱크 방향

**컨텍스트 → 로컬 스토리지 (결정됨)**

매 틱 완료 시점에 `saveAllEntities()` + `save()`를 호출한다. 현재 `TICK_INTERVAL`은 100ms이므로 초당 10회 저장이 발생한다. 성능 부담이 클 경우 N틱 간격(예: 10틱 = 1초)으로 조정한다.

**로컬 스토리지 → 데이터베이스 (논의 필요)**

노트의 DB 전략 옵션을 참고하여 방향을 결정한 후 할 일을 추가한다.

- 틱 처리 위치: `world-context.svelte.ts`의 `startTick()`에서 `setInterval(TICK_INTERVAL)`로 틱 루프 실행.
- `saveAllEntities()`는 월드의 모든 엔티티에 대해 `entity.save()`를 호출하여 런타임 상태(위치, 욕구값 등)를 스토어에 반영한다.
- 관련 파일:
  - 틱 루프: `src/lib/components/app/world/context/world-context.svelte.ts`
  - 스토어 정의: `src/lib/hooks/use-world/use-world.ts`
  - 로컬 스토리지: `src/lib/hooks/use-world/use-world-test-storage.ts`
  - 테스트 월드 초기화: `src/lib/hooks/use-world/use-world-test.ts`
  - 엔티티 save: `world-character-entity.svelte.ts`, `world-item-entity.svelte.ts`, `world-building-entity.svelte.ts`
- DB 싱크 전략 옵션 (논의 필요):
  - **옵션 A: JSON 스냅샷** — 월드 전체 상태를 JSON blob으로 DB에 저장. 테이블 1개로 단순화. 관계형 쿼리 불가하지만 게임 세이브 파일 패턴에 적합. 저장 빈도를 자유롭게 조절 가능.
  - **옵션 B: 현행 관계형 + 배치 저장** — 현재 테이블 구조 유지. 변경사항을 모아서 주기적으로 배치 upsert. FK 관계 유지되어 서버사이드 로직/분석 가능. 테이블 수만큼 쿼리 발생.
  - **옵션 C: 하이브리드** — 관계형 테이블은 초기 세팅/템플릿용으로 유지하고, 런타임 상태는 JSON 스냅샷으로 저장. 두 스키마를 관리해야 하는 복잡도 발생.
  - 고려 사항: 현재 게임은 싱글 플레이어 기반. 멀티플레이어 확장 계획이 있으면 옵션 B가 유리. 싱글 플레이어 전용이면 옵션 A가 가장 단순.
